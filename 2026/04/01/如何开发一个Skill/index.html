打造数字分身：从零构建你的第一个 Claude Code Skill 深度指南

在上一篇文章中，我们深度辨析了 Skill 与 MCP 的本质区别：MCP 是物理工具箱，而 Skill 是教 AI 如何做事的“标准作业程序 (SOP)”。

如果你已经厌倦了每天对着 AI 重复输入“请帮我加上类型注释”、“写完代码记得跑一下 lint”、“提交前按 conventional commits 格式写 message”，那么是时候将这些工作流固化下来了。

本文将带你从零开始，在 Claude Code 中架构并开发一个真正具有生产力的自定义 Skill。

1. 解剖学：一个 Skill 的物理结构

在 Claude Code 中，Skill 并不是一段简单的代码，而是一个自包含的能力包（Capability Package）。
所有的自定义 Skill 都存放在项目根目录（或全局目录）的 .claude/skills/ 文件夹下。

2. 核心引擎：编写 SKILL.md

SKILL.md 是这个技能的大脑。它分为两部分：顶部的 YAML Frontmatter（元数据与 Hooks） 和底部的 Markdown 正文（SOP 逻辑）。

让我们来实现一个名为 CodePolice 的技能，它的作用是在 AI 编写完代码后，强制进行质量审查和规范化。

4. 调试与发布你的 Skill

开发完 Skill 后，如何在终端里使用它？

  1. 隐式唤醒（Agentic 模式）：
    在终端中直接说：“我刚改完订单模块，帮我审查一下代码。”
    Claude 会自动扫描 .claude/skills/，如果发现 code-police 的描述匹配当前任务，它会在回复前自动加载这个技能的 SOP 和 Hooks。

  2. 显式唤醒（强制模式）：
    直接对 Claude 发号施令：“使用 code-police 技能检查当前改动。”

调试技巧

在开发初期，Skill 的表现可能不如预期。你可以使用 claude --dangerously-skip-permissions 配合具体的 prompt 进行反复测试，观察它是如何调用工具和执行 Hooks 的。

5. 结语：从 Prompt Engineering 到 Workflow Engineering

开发一个优秀的 Skill，其核心早已不是过去那种“如何用玄学词汇欺骗大模型”的 Prompt Engineering（提示词工程）。

它本质上是 Workflow Engineering（工作流工程）：你需要像设计微服务一样，思考这个智能体的边界在哪里？哪些逻辑应该用确定性的 Bash 脚本实现？哪些应该用 Hooks 来拦截？哪些才需要真正交给大模型的神经网络去推理？

当你的团队沉淀了十几个这样的高质量 Skill 时，你就真正拥有了一支不知疲倦、永不犯错的数字架构师团队。
