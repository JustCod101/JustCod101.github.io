架构师必修课：分布式事务与最终一致性的深度博弈

在单体应用时代，事务管理是一件极其优雅的事情。一个 @Transactional 注解，数据库的 ACID 特性（原子性、一致性、隔离性、持久性）就能为你包揽一切。无论业务逻辑多么复杂，要么全部成功，要么全部回滚。

然而，当系统演进到微服务架构时，原本在同一个数据库实例中的操作被拆分到了不同的服务、不同的数据库，甚至不同的数据中心。“如何保证跨节点的数据一致性？” 成为了每一位架构师必须面对的梦魇。

本文将带你深入剖析分布式事务的核心流派，并探讨为何“最终一致性”成为了大厂微服务架构的最终归宿。

1. 理论基石：从 CAP 到 BASE

在讨论具体方案之前，我们必须达成理论共识。

1.1 CAP 定理的绝望

CAP 定理指出，一个分布式系统不可能同时满足以下三点：

  • C（Consistency，一致性）：所有节点在同一时间看到的数据完全一致。
  • A（Availability，可用性）：每次请求都能获取到非错的响应（但不保证是最新的数据）。
  • P（Partition tolerance，分区容错性）：由于网络故障导致节点之间失去通信时，系统仍能继续运行。

在分布式网络中，网络分区（P）是必然会发生的客观事实（如光缆被挖断、路由器宕机）。因此，架构师只能在 CP（强一致，放弃高可用，如 Zookeeper）和 AP（高可用，放弃强一致，如 Eureka）之间做抉择。

1.2 BASE 理论的妥协

互联网业务（如电商、社交）对可用性的要求极高。如果因为订单系统和积分系统之间的网络抖动导致用户无法下单，业务是无法接受的。
于是，前辈们提出了 BASE 理论：

  • BA (Basically Available，基本可用)：允许损失部分可用性，保证核心可用。
  • S (Soft State，软状态)：允许系统存在中间状态，且该状态不影响系统的整体可用性。
  • E (Eventual Consistency，最终一致性)：系统保证在一定时间内，数据最终能够达到一致的状态。

微服务架构的指导思想，就是拥抱 AP，追求最终一致性（BASE）。

2. 强一致性流派：2PC 与 TCC

虽然互联网偏爱最终一致性，但在金融、支付等核心链路，依然需要强一致或准强一致的解决方案。

2.1 2PC (Two-Phase Commit, 两阶段提交)

由事务协调者（Coordinator）和参与者（Participant）组成。

  • 第一阶段（准备阶段）：协调者询问所有参与者“可以提交吗？”参与者锁定资源，记录 Undo/Redo 日志，回复 Yes/No。
  • 第二阶段（提交阶段）：如果所有人都 Yes，则协调者发出 Commit 指令；如果有人 No，则发出 Rollback 指令。

痛点：极其笨重。在第一阶段，所有参与者的资源都被锁死，严重降低系统的并发吞吐量。且存在单点故障问题。

2.2 TCC (Try-Confirm-Cancel)

TCC 是应用层的两阶段提交，它不依赖数据库的本地事务锁，而是要求业务方写三个接口：

  1. Try：做业务检查，预留业务资源（如：冻结库存、冻结资金）。
  2. Confirm：确认执行业务操作（如：扣减被冻结的库存）。
  3. Cancel：取消执行，释放预留的资源（如：解冻库存）。

优势：性能极高，因为没有长事务锁。
痛点：对业务的侵入性极大。原来写一个扣库存接口，现在要写三个，还要处理“空回滚”、“悬挂”等极端网络异常场景。目前阿里开源的 Seata 框架对 TCC 有极好的支持。

3. 最终一致性流派：异步与补偿

对于 80% 的业务场景（如：下单后发短信、注册后送积分），我们根本不需要强一致性，只需要保证事情最终办成了就行。

3.1 本地消息表 + 定时任务（经典方案）

如何保证“扣减库存”和“发送 MQ 消息通知积分系统”这两件事要么都成功，要么都失败？

核心思想：将分布式事务转化为本地事务。

  1. 在库存系统的数据库中，除了库存表，再建一张“本地消息表”。
  2. 把“扣库存”和“向消息表插入一条待发送记录”放在同一个 @Transactional 本地事务中。
  3. 如果本地事务成功，说明库存扣了，消息记录也有了。
  4. 后台启动一个定时任务，轮询消息表中的“待发送”记录，投递到 MQ（如 RocketMQ、Kafka）。
  5. 积分系统消费 MQ 消息，增加积分。

痛点：定时任务轮询数据库，存在性能瓶颈，且消息会有延迟。

3.2 可靠消息最终一致性（RocketMQ 事务消息）

阿里将“本地消息表”的思想直接封装到了中间件中。

  1. 业务方向 RocketMQ 发送一条 Half 消息（半消息）。
  2. MQ 收到后，回复 OK。此时这条消息对消费者（积分系统）是不可见的。
  3. 业务方执行本地事务（扣库存）。
  4. 根据本地事务的执行结果，向 MQ 发送 Commit 或 Rollback 指令。
  5. 如果 MQ 迟迟没有收到 Commit/Rollback（网络断了），MQ 会主动回调业务方的接口，反查本地事务的状态。

这套机制完美解耦了业务逻辑与消息可靠性，是目前大厂处理最终一致性的首选方案。

4. Saga 模式：长事务的救星

当你有一个非常长的业务链路（例如：跨国旅游预订：订机票 -> 订酒店 -> 订专车），TCC 的锁定成本太高，本地消息表又难以管理复杂的链路流转。

Saga 模式 的核心思想是将一个长分布式事务拆分为一系列本地事务 $T_1, T_2, …, T_n$。
同时，业务方需要为每一个本地事务提供一个补偿操作 $C_1, C_2, …, C_n$。

  • 正常流程：$T_1 \to T_2 \to T_3$ 依次成功。
  • 异常流程：如果 $T_3$ 失败了，Saga 协调器会自动反向执行补偿动作：$C_2 \to C_1$。

Saga 相比于 TCC，没有 Try 的预留阶段，它直接执行真实的业务操作。如果后面报错了，再通过“冲正”（比如把扣掉的钱退回去）来弥补。这非常符合现实世界中的商业逻辑。

5. 总结：没有银弹，只有权衡

在架构设计的世界里，所有高大上的概念都是为了应对残酷的物理现实。

  1. 核心金钱交易：能用单体事务就用单体事务，逼不得已跨库时，老老实实写 TCC 代码。
  2. 边缘非核心链路：坚决拥抱 MQ 事务消息/本地消息表，追求最终一致性。
  3. 复杂编排链路：引入 Saga 状态机。

终极法则：最好的分布式事务，就是没有分布式事务。在进行微服务拆分时，尽量根据领域驱动设计（DDD），将需要强一致性的数据划分在同一个服务和同一个数据库中。合并领域，往往比引入复杂的分布式事务中间件要聪明得多。
