突破 RAG 瓶颈：从朴素检索到高级检索（Advanced RAG）的优化策略

在 LLM 应用开发中，RAG（Retrieval-Augmented Generation，检索增强生成） 已经成为解决大模型“幻觉”和“知识知识滞后”的行业标准。

然而，当你用 LangChain 或 LlamaIndex 写完一个“读取 PDF -> 文本切分 -> 存入 ChromaDB -> 向量相似度搜索”的 Demo 后，你会绝望地发现：这套“朴素 RAG”在生产环境中的表现简直惨不忍睹。
它要么搜不到关键信息，要么把大量不相关的冗余文本塞给大模型，导致回答跑偏。

本文将带你走出“新手村”，深度解析构建 Advanced RAG 的核心优化策略。

1. 召回阶段优化 (Pre-Retrieval)

“垃圾进，垃圾出”。如果存入向量数据库的数据本身就是支离破碎的，检索效果绝对好不了。

1.1 文档分块策略 (Chunking Strategy)

不要盲目使用按字数硬切分（如每 500 字一刀）。这会把一段完整的语义劈成两半。

  • 语义分块 (Semantic Chunking)：利用 NLP 技术识别段落、标题边界，或者计算相邻句子的余弦相似度，当相似度突变时进行切分，保证每个 Chunk 都是一个完整的语义单元。
  • 重叠切分 (Overlap)：即使是基于字数，也必须设置 chunk_overlap（例如 500字 Chunk，100字 Overlap），确保上下文的连贯性。

1.2 元数据注入 (Metadata Filtering)

单纯的向量相似度搜索是很盲目的。
优化方法：在存入 Chunk 时，一定要打上结构化标签（如：date: 2023-10，author: 张三，doc_type: 财务报表）。在用户查询时，先通过规则或 LLM 提取条件，进行 Metadata 预过滤（Pre-filtering），然后再在过滤后的子集中进行向量搜索。这能极大提升检索的准确率和速度。

2. 检索阶段优化 (Retrieval)

用户的提问往往很短（比如：“去年 Q3 的营收为什么下降？”），直接用这句短话去跟数据库里长篇大论的文档算相似度，效果极差。

2.1 查询重写与扩展 (Query Transformation)

在去向量数据库搜索前，先让 LLM 对用户的 Query 动点手脚：

  • Query Rewrite：纠正用户的错别字，补全上下文。
  • Multi-Query：让 LLM 根据原问题，生成 3-5 个不同表述的相似问题。用这 5 个问题分别去检索，然后将召回的文档取并集。这能有效对抗用户表达与文档表述之间的“词汇鸿沟”。
  • HyDE (Hypothetical Document Embeddings)：这是个极其天才的想法。面对用户提问，先让 LLM “凭空捏造” 一份假答案。虽然假答案可能包含幻觉，但它的语义结构和行文模式一定与真实的参考答案极其相似。用这篇“假答案”的向量去数据库里搜，往往能召回极其精准的文档。

2.2 混合检索 (Hybrid Search)

单纯的向量搜索（Dense Retrieval）对“专有名词”、“产品序列号”等精准匹配非常无力。
优化方法：向量检索（语义相似） + 关键词检索（BM25 / TF-IDF，精准字面匹配）。
几乎所有的现代向量数据库（如 Milvus, Qdrant, Pinecone）都支持混合检索。两种结果召回后，通过一定权重相加即可。

3. 排序与生成优化 (Post-Retrieval)

从数据库里捞出了 20 个相关的 Chunk，如果一股脑全塞进 Prompt 里，大模型会遭遇“中间上下文丢失（Lost in the Middle）”问题。

3.1 重排序 (Reranking)

向量模型（Embedding Model）为了追求速度，往往会损失一定的精度。
优化方法：使用专门的 Cross-Encoder（交叉编码器，如 BGE-Reranker 或 Cohere Rerank）。

  • 步骤 1：通过便宜极快的向量搜索，召回 Top 100 粗筛结果。
  • 步骤 2：用 Rerank 模型，把 User Query 和这 100 个结果两两配对，进行深度注意力计算，打出精准的匹配分。
  • 步骤 3：只取重排后得分最高的 Top 5 喂给生成模型。

3.2 父文档检索 (Parent-Child Retriever / Auto-merging)

这是一个经典的矛盾：Chunk 越小，向量匹配越精准；但 Chunk 越小，塞给大模型生成的上下文越匮乏。
优化方法：
存入数据库时，把长文档切成大块（Parent Chunk），再把大块切成小块（Child Chunk）。
搜索时：匹配小块（精准命中），提取大块（丰富上下文）喂给 LLM。
这既保证了检索的命中率，又保证了生成的质量。

4. 终极架构：Agentic RAG

当系统演进到最后，RAG 不再是一个线性的管道（Pipeline），而会变成一个 Agent。
在 Agentic RAG 中，系统会根据用户的复杂提问，自主判断是否需要检索、使用哪个向量库、是否需要多次跳跃检索（Multi-hop QA），甚至在生成回答后自己做一遍事实验证（Self-Reflection），如果发现幻觉，就修改 Query 重新检索。

结语：
RAG 的本质是一场信息学的“大海捞针”。从朴素 RAG 到 Advanced RAG，核心思想只有一个：不要迷信向量相似度，用工程化的手段（分块、重写、混合检索、重排）去弥补单纯数学计算的不足。
