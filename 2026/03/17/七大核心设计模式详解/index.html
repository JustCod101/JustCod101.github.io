内化于心：日常开发中最核心的 7 种设计模式

GoF（四人帮）总结了 23 种设计模式，但说实话，有些模式你可能职业生涯都用不上几次。根据二八定律，掌握这 7 种设计模式，就能解决日常开发中 80% 的代码坏味道。

为了让你“过目不忘”，我拒绝晦涩的 UML 图，直接用最直白的生活隐喻和一句话本质，帮你将它们刻在脑子里。

1. 策略模式 (Strategy) —— “随时换脑”

一句话本质：将不同的算法（策略）封装成独立的类，让它们可以互相替换，从而消灭成堆的 if-else。

生活隐喻：
你（Context）要去机场。你可以选择打车（策略 A）、坐地铁（策略 B）或骑共享单车（策略 C）。你的目的都是“到机场”，但具体怎么去，可以在出发前临时决定。

什么时候用？
当你发现代码里出现了 if (type == "支付宝") { ... } else if (type == "微信") { ... } else if (type == "银联") { ... } 这种又臭又长的判断时，立刻使用策略模式，把每种支付方式写成一个独立的类，实现同一个 PaymentStrategy 接口。

2. 观察者模式 (Observer) —— “微信公众号”

一句话本质：一对多的依赖关系，当一个对象状态改变时，所有依赖它的对象都会得到通知并自动更新。

生活隐喻：
你关注了一个微信公众号（Subject / 被观察者）。当公众号发新文章时，系统会自动把文章推送到你的手机上（Observer / 观察者）。你不需要每隔五分钟去刷新一下公众号看看有没有更新（告别低效的轮询）。

什么时候用？
经典的 MVC 架构（Model 变了，View 自动更新）、前端的事件监听（onClick）、后端的 MQ 消息广播订阅，本质上全都是观察者模式。

3. 装饰器模式 (Decorator) —— “游戏角色穿装备”

一句话本质：在不改变原有对象代码的情况下，动态地给对象增加新的功能（比继承更灵活）。

生活隐喻：
你创建了一个游戏角色（基础对象）。
你给他穿上了一件盔甲（装饰器 A），他有了防御力。
你又给他配了一把屠龙刀（装饰器 B），他有了攻击力。
盔甲和屠龙刀并没有改变“角色”的本质，只是“包裹”在角色外面，附加了新能力。

什么时候用？
当你想要给一个类扩展功能，但不想写一堆子类（避免“子类爆炸”）时。比如 Java 的 I/O 流：new BufferedInputStream(new FileInputStream(file))，一层套一层，极致优雅。

4. 工厂方法模式 (Factory Method) —— “代工厂”

一句话本质：定义一个创建对象的接口，但让子类决定实例化哪一个类。将 new 操作延迟到子类。

生活隐喻：
苹果公司（抽象工厂）只制定设计规范，不负责亲自组装手机。
富士康（具体工厂 A）负责生产 iPhone。
立讯精密（具体工厂 B）负责生产 AirPods。
你作为消费者，只需要找苹果公司“买”产品，不需要知道具体是哪个厂子、用什么螺丝拧出来的。

什么时候用？
当一个类的实例化过程非常复杂，或者你需要根据不同的条件创建不同的实例，但又不想把一堆 new 操作硬编码（Hardcode）在业务逻辑里时。

5. 单例模式 (Singleton) —— “唯一真神”

一句话本质：确保一个类在整个系统中只有一个实例，并提供一个全局访问点。

生活隐喻：
一个国家只能有一个现任总统。无论谁去问“总统是谁”，得到的答案都必须是同一个人。

什么时候用？
非常明确的全局共享资源，比如：数据库连接池、全局配置管理器（Config）、操作系统的任务管理器。
注：一定要注意多线程并发情况下的线程安全问题（如：双重检查锁 Double-Checked Locking）。

6. 适配器模式 (Adapter) —— “电源转换头”

一句话本质：将一个类的接口转换成客户希望的另一个接口，使得原本由于接口不兼容而不能一起工作的类可以一起工作。

生活隐喻：
你带着国行版的笔记本电脑去了欧洲，发现插座是圆孔的（旧接口），而你的插头是扁平的（新需求）。怎么办？买一个“电源转换头（Adapter）”插在中间。

什么时候用？
你引入了一个极其好用的第三方库，但是它提供的函数方法名和你系统里现有的完全对不上。你不想改自己的系统，也改不了第三方库的源码。这时，写一个适配器类，包装一下第三方库，向外暴露你系统认识的接口。

7. 外观模式 (Facade) —— “前台接待处”

一句话本质：为子系统中的一组接口提供一个一致的、更高层的界面，使得子系统更加容易使用。

生活隐喻：
你去医院看病。如果你自己去，你需要挂号、找诊室、化验、交费、取药，在不同科室之间跑断腿（调用各种底层子系统）。
但如果你去了一家高端私立医院，会有个“私人管家（Facade）”全程跟着你，你只需要对管家说“我要看病”，他就会帮你把所有繁琐的流程都走完。

什么时候用？
当你有一个极其复杂的庞大系统时。比如启动一辆汽车，底层涉及到点火系统、喷油系统、起动机等十几个模块，但暴露给驾驶员的只有一个简单的“一键启动”按钮。

💡 终极心法：超越模式

记住，设计模式不是法律，而是词汇表。

我们学习设计模式，不是为了在代码里生搬硬套（过度设计是万恶之源），而是为了在和同事沟通时，能够用最简洁的语言达成共识：

  • “这里太乱了，我们抽个策略出来吧。”
  • “这个老接口别改了，包个适配器吧。”

当这七个词汇融入你的直觉，你的代码品味就已经超越了 80% 的同行。
